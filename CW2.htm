Visual Scripting Cafe Application Architecture

A software architecture analysis of the cafe simulation application built using Visual Scripting in Tutorials 08/09 and 10.

1) Application Overview

The application is a Unity cafe simulation game where a player controls a character (FPC) who operates a cafe by managing orders, preparing burgers, serving diners, and collecting payment.

High-level gameplay flow:

  1. Player exploration: First-person controller navigates the cafe environment.
  2. iPad interface: World-space UI canvas attached to player shows cafe status (timers, inventory counts).
  3. Diner spawn/ordering: Customer (Annie NPC) appears and orders via dialogue.
  4. Order management: Player uses iPad to track cook timers and burger inventory.
  5. Service: Diner prefab shows UI panel with order state, player delivers burger and order.
  6. Eating and payment: Diner eats and pays, generating revenue and happiness score.
  7. Loop repeat: Next customer spawns, cycle continues.

Technology stack:

  • Engine: Unity
  • Scripting method: Bolt Visual Scripting (node-based, no C# coding required)
  • UI framework: Unity UI (Canvas, buttons, text displays)
  • Character system: Animator with pre-built sitting/animation loops

Key insight: This architecture demonstrates building a complete gameplay loop without traditional C# scripting, relying entirely on visual node graph composition.

2) Core Components and Responsibilities

A) Player/Camera Subsystem

  • Component: First-Person Controller (FPC)
  • Responsibility: Character movement, camera control, input handling.
  • Scope: Entire scene navigation.

B) iPad Interface Subsystem

  • Components: World-space Canvas (child of player capsule), UI text elements, buttons.
  • Responsibility: Display cafe status, timers, burger inventory (uncooked/cooked counts), order workflow UI.
  • Scene variables tracked: BurgersUncooked, BurgersCooked, OnBoard, timers.
  • Interaction method: Buttons triggering visual script events.

C) NPC / Diner Subsystem

  • Component: Annie prefab (character model with animator).
  • Responsibility: Represent customer, play dialogue, animate, track satisfaction/happiness.
  • Interaction points: Dialogue trigger, order delivery UI panel.

D) Diner UI Prefab Subsystem

  • Components: World-space canvas panel containing buttons (Dialogue, Order, Deliver, Payment), text displays, dialogue box.
  • Responsibility: Present customer interaction options, show order state, display dialogue, handle payment confirmation.
  • Data flow: Receives order info from parent script, displays timers and state feedback.

E) Burger/Item Prefab Subsystem

  • Component: Burger game object with pickup button.
  • Responsibility: Represent food item, handle pickup interaction via OnClick button.
  • Scene variables: OnBoard (tracks if burger is available), BurgerUpBool (visibility toggle).

F) Environment / World Objects

  • Components: Cafe (static environment), chairs, tables.
  • Responsibility: Provide spatial context and seating areas for NPCs.

G) Scene Variables / Global State

  • Purpose: Central storage for gameplay state accessible across prefabs and scripts.
  • Key variables: BurgersUncooked, BurgersCooked, OnBoard, BurgerUpBool, Happiness, CookTime, OrderWaitTime.
  • Pattern: Shared state allows decoupled communication between iPad, diner, and burger systems.

3) Control Flow and Data Interactions

Initialization sequence:

  1. Scene loads; iPad canvas and scene variables initialize.
  2. VS Manager (root controller) sets up initial state.
  3. First customer (Annie prefab) spawns at diner location.

Interaction sequence per customer:

  1. Dialogue phase: Annie prefab displays "Dialogue" button in UI panel. Player clicks to view customer dialogue. Ren_Order script in visual scripting plays dialogue sequence.
  2. Order phase: Dialogue triggers iPad to display "Order" button. Visual script updates order expectations in scene variables.
  3. Cooking phase: iPad timer script counts down. When complete, burger state changes from Uncooked to Cooked. Burger prefab appears with "Pickup" button.
  4. Delivery phase: Player clicks iPad's "Deliver Order" button or clicks diner's UI to move to table. Visual script triggers animation and waiting state in TimeScale.
  5. Payment/Finish: After eating timer expires, "Payment" button appears on diner UI. Player clicks to complete transaction. Happiness value determined. Money/tips added to account. Diner prefab destroyed.
  6. Respawn: Next customer spawns, loop repeats.

Visual scripting node structure:

  • iPad Canvas graphs: Separate graphs for Timer, SpawnButton, Order, CookTimer, SpawnBurger, TextUpdate.
  • Annie Prefab graphs: Dialogue, Order state, WaitTimer, EatTimer, PaymentTimer, Happiness calculation, Destroy logic.
  • Control flow: Graphs communicate via scene variable updates and OnClick events. Event listeners trigger downstream behavior.

4) Architectural Patterns and Strengths

Strengths present in this design:

  • Component-based prefabs: iPad, Annie, Burger are self-contained prefab units with their own canvases and visual script graphs. Easy to redesign or reuse.
  • Scene variables as shared state: Decouples systems; iPad logic and diner logic don't directly reference each other, only update shared variables.
  • Visual scripting accessibility: No C# knowledge required; designers and students can build logic using drag-and-drop nodes.
  • Clear interaction points: Buttons and UI elements define player actions explicitly.
  • Timers and animations: Separate visual script graphs for timing systems keep flow readable.
  • Prefab instantiation pattern: Annie (or burger) spawning uses prefabs, enabling easy customer variation in future iterations.

Architectural observations:

  • Monolithic scene variables: As customer count grows, shared global variables become hard to manage. Each customer might need its own state context.
  • iPad as central UI: A good affordance (world-space UI on player), though functionality could eventually split into subsystems if complexity grows.
  • Direct event-to-action mapping: Button clicks directly update variables and trigger animations. Simple and effective for small projects, but lacks middle layer abstraction for complex workflows.
  • Time-based state: Reliance on coroutines and timers (WaitTimer, CookTimer, PaymentTimer) works well for sequential interactions.

5) Potential Improvements and Scalability

Scaling challenges (when extending beyond single-customer loop):

  1. Multiple simultaneous customers: Scene variables cannot track state for multiple diners at once. Solution: Create a customer/order data structure or use prefab-level variables instead of global scene variables.
  2. Visual script graph complexity: As logic grows, individual graphs become large and hard to navigate. Solution: Break graphs into sub-graphs (encapsulated visual script units) for reusability.
  3. Burger queue management: Current design assumes instant burger preparation. Real queue system requires tracking multiple burger orders. Solution: Maintain an order queue data structure or use spawned burger prefabs with unique IDs.
  4. Diner queue and table management: Extending to multiple tables requires assigning customers to specific tables and managing occupancy. Solution: Implement a table registry or use Inspector-assigned table references.

Best-practice recommendations:

  • Break visual scripts into reusable sub-graphs: Create a "Customer Order Manager" sub-graph that handles one diner's lifecycle, then instantiate copies for multiple customers.
  • Use prefab-local variables: Instead of relying on scene variables, store customer-specific state (order ID, satisfaction) as variables on the prefab itself.
  • Introduce a game manager / flow controller: A central visual script that coordinates customer spawning, table assignment, and order routing.
  • Separate UI logic from game logic: Create a dedicated iPad UI controller graph that only reads from shared state, leaving gameplay logic in prefabs.
  • Document graph intent: Larger projects benefit from clear naming conventions and comments within visual script nodes.

Visual scripting maturity path:

  • Prototype stage (current): Single-customer loop with centralized scene variables. Good for learning and rapid iteration.
  • Alpha stage: Multiple customers, table assignment, queue visualization. Requires refactoring toward data-driven architecture.
  • Production-ready: Stable multi-customer flow, robust error handling, performant graph execution, modular sub-graphs.

6) Key Takeaways

This cafe simulation demonstrates visual scripting as a valid game development approach for prototypes and educational projects:

  • Strengths: Accessible to non-programmers, rapid iteration, visual clarity of logic flow, prefab-based component design.
  • Limitations: Graph complexity grows quickly; shared global state becomes unwieldy with scale; performance overhead of node interpretation at runtime.
  • Best practices: Keep individual graphs focused; use sub-graphs for reusability; manage state at prefab level when possible; think in terms of systems and communication.
  • Scalability cues: When rules become difficult to express in visual nodes, or when adding a 10th simultaneous customer breaks architecture, consider moving core logic to C# while using visual scripting for UI/event orchestration.

Conclusion: This project is a strong example of visual scripting applied effectively to a game loop. The architecture works well for its scope; thoughtful refactoring (sub-graphs, prefab-level state, a flow controller) would enable scaling to larger, multi-table scenarios while maintaining visual-script-only development.
