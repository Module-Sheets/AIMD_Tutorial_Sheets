Visual Scripting Cafe Application Architecture

1) Purpose & Scope

This system is a Unity 3D prototype where a player (First Person Controller) operates in a café scene and completes an interaction loop with a diner prefab (Annie). Interaction is driven by:

  • a player-attached iPad with World Space UI to start gameplay and run "kitchen" actions (cook/spawn burger, show timers/counters), and
  • a diner prefab UI panel (World Space) that progresses through a staged sequence: Dialogue → Order → Deliver Order → Take Payment → Destroy/leave, with timers and happiness/score feedback.

2) System Context

Actors / external dependencies

  • Player: navigates scene and triggers UI actions.
  • Unity Engine: rendering, physics, UI, prefab instantiation/destruction.
  • Starter Assets (FPC): movement/camera controller package.
  • Unity Visual Scripting: primary logic implementation (graphs, variables, script machines).
  • TextMeshPro (optional/likely): used for improved text rendering; requires Visual Scripting node type options regeneration if used.
  • Imported content: Café model, chair, iPad model (or cube proxy), burger/beer/fries, Annie character, sitting loop animations (Mixamo/Fuse pipeline).

3) Architectural Overview (High-Level)

Key design decisions

  1. Diegetic World Space UI: UI exists "in-world" (on the iPad and on the diner panel), not as screen overlay.
  2. Logic in Visual Scripting graphs: timers, button flows, variable updates, prefab destruction are graph-driven.
  3. Two-tier state model:
    • Scene Variables for global/shared gameplay state (inventory, cook timers, onboard states, overall feedback).
    • Prefab Object Variables for per-diner local state (interaction stage, per-diner timers, dialogue counters, local UI refs).
  4. Prefab modularity: Annie diner is packaged as a prefab with animator + UI + graphs + variables; enabling future scaling to multiple diners.

4) Logical Architecture (Components and Responsibilities)

4.1 Scene Composition

A) Player (First Person Controller)

  • Provides locomotion and camera.
  • Can be toggled on/off (e.g., "Start Disable" behaviour) via a Visual Scripting graph (often hosted by a manager object).

B) iPad (3D object)

  • Parent/attached to player capsule.
  • Uses an iPad root transform to normalize transform/pivot/scaling (especially if using cube proxy and/or if parent is scaled).

C) iPad Canvas (World Space UI)

  • Renders timer and "value display" texts.
  • Hosts buttons that trigger global actions (start, cook, spawn burger, update display).
  • Has Script Machine / Embedded graphs implementing:
    • timer countdown logic,
    • inventory/counter updates (uncooked/cooked),
    • burger spawning / "onboard" state,
    • periodic UI refresh.

D) GameManager (empty GameObject + Visual Scripting graph)

  • Provides an always-present "host" graph for scene-wide control.
  • Useful for:
    • initialization,
    • enabling/disabling FPC,
    • maintaining/setting Scene Variables,
    • orchestrating basic scene functions.

E) Environment (café container + chair + colliders)

  • Static meshes and colliders (stairs colliders explicitly noted).

4.2 Prefab Architecture

A) Annie Diner Prefab — Contains:

  • Annie character model (humanoid rig).
  • Animator Controller with sitting/eating loop animations.
  • Diner UI panel (World Space canvas) with:
    • Dialogue area + dialogue counter,
    • Buttons in prescribed order:
      1. Dialogue
      2. Order
      3. Deliver Order
      4. Take Payment
  • Visual Scripting graphs implementing the diner state machine and timers.
  • Prefab Object Variables capturing per-instance state.

B) Burger (and other food props)

  • Spawned and tracked via Scene Variables.
  • Tutorial 10 notes burger has no local variables and uses Scene Variables such as OnBoard and BurgerUpBool.
  • Includes a temporary pickup button to validate onClick wiring and show/hide behaviour.

5) Data Architecture (State, Variables, and Bindings)

5.1 Scene Variables (global)

Used by iPad UI graphs, burger logic, and referenced by diner logic when needed.

Typical categories:

  • Cook timer state (countdown, ready state).
  • Inventory counters:
    • uncooked burger supply,
    • cooked burger count/tracker.
  • Carry/availability state:
    • "OnBoard" (whether burger is held/available for delivery),
    • "BurgerUpBool" (visibility/pickup toggle).
  • Global feedback:
    • happiness/score summary to show on iPad.

UI binding pattern: "Value displays" are created by duplicating label texts, renaming them (e.g., "Value"), and having graphs set their text based on scene variables.

5.2 Prefab Object Variables (local)

Stored on each diner prefab instance, typically including:

  • interaction stage flags (dialogue done, order placed, delivered, payment due),
  • timers: order wait, delivery wait, eat timer, pay timer,
  • dialogue index/counter,
  • references to UI Text/TextMeshPro elements and buttons,
  • references for destroy/remove behaviour.

6) Behavioural Architecture (Key Flows)

6.1 Start / enable player control

  1. Player presses Start on iPad UI.
  2. GameManager graph toggles FPC enabled state.
  3. Scene Variables initialize/reset (as needed).

6.2 Cook and spawn burger (global loop)

  1. Player triggers cook timer on iPad UI.
  2. Timer counts down; iPad text updates each tick.
  3. At completion:
    • uncooked supply decrements (Tutorial 10 "uncooked update"),
    • cooked burger tracker increments/updates.
  4. Player triggers "Spawn Burger":
    • checks cooked availability,
    • updates onboard / carry booleans,
    • spawns or enables burger object.

6.3 Diner interaction loop (per prefab)

  1. Dialogue: advances dialogue text/counter.
  2. Order: begins order-wait timer.
  3. Deliver: delivery-wait + eat timer sequences; may check global onboard/cooked state.
  4. Payment: pay timer stage; updates happiness; triggers destroy (removes prefab instance).
  5. iPad receives updated happiness state via Scene Variables and reflects it in UI.

7) UI Architecture

7.1 World Space UI rationale

  • Supports diegetic UI: player "looks at" iPad and diner panel in the 3D world.
  • Requires careful transform normalization:
    • iPad root empty object,
    • avoid inheriting problematic scaling from parent objects.

7.2 Text system choice

  • Standard Unity Text or TextMeshPro UGUI.
  • If TextMeshPro is used, Visual Scripting node types must be enabled and nodes regenerated.

8) Deployment / Packaging View (Unity Project Organisation)

Recommended structure implied by tutorial workflow:

  • Assets/Models/ (Cafe, Chair, iPad, Annie)
  • Assets/Textures/ (café sign/posters/iPad screen images)
  • Assets/Animations/ (sitting loops)
  • Assets/Prefabs/ (Annie prefab, burger prefab, UI prefabs if any)
  • Assets/VisualScripting/ (graphs, macro assets if extracted)
  • Assets/Scenes/ (main café scene)

9) Quality Attributes & Considerations

Maintainability

  • Pros: Prefab encapsulation + graphs encourage modular thinking.
  • Cons: Visual Scripting graphs can become hard to version/diff at scale; naming conventions for variables and graphs become critical.

Scalability (functional)

  • Architecture supports multiple diners by instantiating multiple prefab instances.
  • Key risk: overuse of Scene Variables for per-diner state can cause cross-talk; per-diner data should remain in Prefab Object Variables.

Usability

  • Clear button ordering and staged timers guide the player.
  • Value displays on iPad provide continuous feedback (inventory/timers/happiness).

Performance

  • For small numbers of prefabs and simple graphs, performance is fine.
  • For many diners, consider throttling UI updates (avoid updating many text nodes every frame).

10) Risks & Mitigations
  1. Transform scaling/pivot issues (iPad attachment)
    • Mitigation: root empty object normalization; avoid scaled parents; add intermediate parent if needed.
  2. TextMeshPro node availability in Visual Scripting
    • Mitigation: enable TMP types in Visual Scripting project settings; regenerate nodes; standardize on one text system early.
  3. State coupling via Scene Variables
    • Mitigation: keep global-only values in Scene Variables; keep per-diner state in prefab variables; use consistent naming.
  4. Prefab destruction references
    • Tutorial note: Destroy unit targets the Annie prefab; ensure correct target reference per instance.

11) Extension Points (Future Work)
  • Multiple diners with shared queue/list (Tutorial 10 mentions diner list for extra marks).
  • More robust item interaction (replace pickup button with raycast/interact system).
  • More formal state machine pattern per diner (still possible in Visual Scripting, but with clearer state enum).
  • Save/load of global counters (optional).
  • Better separation: move global "kitchen" logic into a dedicated KitchenManager graph.

12) Appendix: Key Components Checklist
  • FPC imported and working
  • iPad model/cube attached to player + normalized root
  • iPad World Space canvas aligned to screen
  • Scene Variables created (inventory/timers/onboard/happiness)
  • GameManager empty object hosts global VS graph
  • Annie imported, humanoid rig configured, sitting loops set up
  • Annie prefab created with UI panel + graphs + prefab variables
  • Diner interaction sequence implemented with timers + destroy
  • Burger spawn + onboard state + pickup/delivery plumbing
