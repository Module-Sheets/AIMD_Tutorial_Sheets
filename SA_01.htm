1) Overall functionality (from a software architecture viewpoint)

This project is a Unity café simulation loop built around:

  • A central game-state/controller: CafeManager
    • Tracks completion flags like restocking, shop open, back door state, music, etc.
    • Triggers progression (e.g., after tasks are done, spawn NPCs / open door / etc.).
  • Interaction as input events on scene objects: InteractionHandler
    • Unity-style: clicking objects (OnMouseDown) triggers actions by calling CafeManager.ToggleTaskCompletion(...) or similar.
  • NPC lifecycle pipeline:
    • NPCSpawner (singleton-ish via static instance) spawns random NPC prefabs and injects their waypoint routes / table assignments via TableData.
    • Ren_Waypoint moves NPCs along waypoint routes to reach interaction points / tables and can enable dialogue.
    • Ren_Order runs an order dialogue sequence (text, audio, animation states) and connects to the till for ordering.
    • EatAndLeave handles the “consume at table → leave → possibly spawn next NPC” loop, and awards tips.
  • Ordering + UI orchestration: Till
    • Owns UI widgets (buttons, text fields) and the order screen state machine (open/close, add items, confirm/cancel).
    • Calculates totals and updates “earned money” and “earned tips”.
  • Table as an in-world domain object: Table
    • Stores what food/drink IDs are expected/placed, holds transforms for placements, and can clean/reset.
    • Resets external systems like BewerageMaker.
  • Door subsystem:
    • Door animates/moves a door.
    • DoorTrigger listens for physics trigger (OnTriggerEnter) and opens the door when an NPC enters.

Architectural style (as implied by the diagram):

  • Predominantly MonoBehaviour-driven, object-oriented, with runtime references between components.
  • A blend of GameObject composition (Unity components) and “manager” orchestration (CafeManager, NPCSpawner, Till).
  • Gameplay is mostly event-driven (clicks, triggers, button presses) but implemented through direct method calls between components.

2) Detailed architecture explanation (for university students) + best practice framing

A. Identify subsystems (good architectural habit)

A clean way to reason about this project is to split it into subsystems:

  1. Game State / Progression
    • CafeManager
    • Responsibility: maintain “what tasks are done” and decide what can happen next.
  2. Interaction/Input
    • InteractionHandler
    • Responsibility: translate player actions (click) into domain actions (“restock completed”, “open back door”, etc.).
  3. NPC / AI Flow
    • NPCSpawner, Ren_Waypoint, EatAndLeave
    • Responsibility: create NPCs, route them, manage their lifecycle.
  4. Narrative / Dialogue + Feedback
    • Ren_Order, DialogueLine, OrderData
    • Responsibility: show lines, play audio, set animation states, advance dialogue.
  5. Ordering / Economy / UI
    • Till, Item
    • Responsibility: UI flow + pricing + payouts.
  6. World Objects
    • Table, Door, DoorTrigger (+ referenced BackDoorAnim, BewerageMaker)
    • Responsibility: in-world state and animations.

This is a “software architecture lens”: what changes together belongs together. If you change UI, you ideally shouldn’t break NPC movement, etc.

B. Key architectural relationships (and what they imply)

1) CafeManager as a “God object” risk

The diagram shows many systems calling into or being triggered by CafeManager (interaction, restock, spawner, back door animation). This is common in student Unity projects.

Best practice idea: keep CafeManager as a state holder + coordinator, but avoid letting it become the place where “everything happens”. When a single class grows, it becomes:

  • hard to test,
  • hard to reuse,
  • hard to understand without reading the entire codebase.

2) Direct references create tight coupling

Examples from the diagram:

  • InteractionHandler --> CafeManager
  • Ren_Order --> Till
  • EatAndLeave --> NPCSpawner (spawns another NPC)
  • Till --> EatAndLeave (tells NPC to move to table)
  • Table --> BewerageMaker

This is tight coupling: changing one class often forces changes in the other.

Best practice: introduce interfaces or an event system for communication.

  • UI can raise OrderConfirmed(foodId, drinkId) event
  • NPC can subscribe to it if it’s the current customer
  • Or a mediator/controller coordinates.

3) UI and domain logic appear mixed in Till

Till owns buttons, text fields, but also pricing, items list, tips, totals, and triggers NPC actions.

Best practice separation:

  • View (UI): just shows/hides screens, reads button clicks.
  • Controller/Presenter: processes UI events, updates view.
  • Domain (Model): order calculation, price list, validation.

In Unity you can still do this with MonoBehaviours, but keep the responsibilities clean.

4) Data structures (OrderData, DialogueLine, Item) are good signs

These are simple data carriers. That’s a good step toward a cleaner architecture because they can become:

  • ScriptableObjects (Unity-friendly data assets),
  • serializable DTOs,
  • testable logic inputs.

C. A “cleaner” target architecture (teachable version)

If I were explaining an idealised architecture for this project to students:

  • Core domain layer (pure C# if possible)
    • Order, MenuItem, PriceCalculator, DialogueSequence
    • No MonoBehaviour, no Unity UI references.
  • Application layer (game rules)
    • CafeSessionController (coordinates tasks, NPC queue, order flow)
    • Emits events like OnCustomerArrived, OnOrderTaken, OnOrderServed.
  • Presentation layer
    • TillView (buttons/text), DialogueView
    • Subscribes to controller events, forwards button clicks to controller.
  • Infrastructure layer (Unity specifics)
    • NPCSpawner, movement components, audio, animation
    • Adapters that translate controller intent into Unity actions.

You don’t have to fully implement Clean Architecture, but thinking in layers helps keep code maintainable.

3) Suggestions / refactoring opportunities (based on the diagram)

High-impact refactors

  1. Replace NPCSpawner.instance with dependency injection or scene references
    • static instance is quick, but encourages global access.
    • Better: assign references in the Inspector, or have a single GameContext object that holds services.
  2. Introduce an event bus (UnityEvent/C# events)
    • Instead of Till -> EatAndLeave direct call:
      • Till raises OrderConfirmed(foodId, drinkId)
      • The active NPC (or a controller) responds.
    • This reduces bidirectional coupling.
  3. Split Till into UI and logic
    • TillView (buttons/text)
    • TillController (order screen flow, invokes services)
    • OrderService or OrderCalculator (prices, totals, validation)
    • This makes it far easier to test and reason about.
  4. Use ScriptableObjects for menu items and dialogue
    • Item could become MenuItemSO (name, price, id, prefab refs).
    • OrderData could be a ScriptableObject asset representing a customer script/dialogue.

Medium-impact improvements

  1. Define explicit NPC state machines
    • Many Unity NPC scripts grow messy with booleans (canMove, isLastWaypointReached, etc.).
    • Better: an enum like CustomerState = Arriving, Ordering, WaitingForFood, Eating, Leaving.
    • Even better: a small state pattern.
  2. Avoid stringly-typed task toggles
    • ToggleTaskCompletion(taskName: string) is fragile (typos break logic).
    • Use an enum TaskType or separate explicit methods/events.
  3. Clarify ownership of “money/tips”
    • Currently Till seems to track earnings/tips.
    • Consider a Wallet/EconomyManager that owns totals; UI displays it.
