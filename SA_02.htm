Overall functionality (architecture-level summary from the diagram + the fetched Assets/Scripts code)

At a high level, this Unity project implements a single-customer gameplay loop for a café:

  1. Spawn a customer (NPC) at a spawn point (NPCSpawner).
  2. The NPC walks along entry waypoints to a dialogue location (Ren_Waypoint).
  3. The NPC plays a dialogue sequence (text + optional audio + animation) and then sets an expected foodID/drinkID order (Ren_Order).
  4. The Till UI lets the player select exactly two items (one food, one drink) and validates whether the chosen pair matches the NPC’s expected IDs (Till).
  5. If correct, the NPC is instructed to walk to their table and sit, then later eat and leave, during which they may tip the player (EatAndLeave + Till.Tip(...)).
  6. When leaving, the NPC triggers spawning the next NPC, then destroys itself (EatAndLeave calls NPCSpawner.instance.SpawnRandomNPC()).
  7. Tables can accept placed food/drink objects and be cleaned/reset (Table), and doors can open via trigger collisions (Door, DoorTrigger).

From a software architecture viewpoint, this is a scene-driven, component-based architecture typical of Unity projects, where MonoBehaviours act as both controllers and domain objects, and where gameplay is coordinated through:

  • direct component references (GetComponent, public fields), and
  • scene queries (FindObjectOfType, FindAnyObjectByType), and
  • a global-ish singleton (NPCSpawner.instance).

Detailed software architecture explanation (for university students), highlighting best practice

1) Identify the main subsystems (separation of concerns)

You can understand this project as a small set of cooperating subsystems:

A. NPC lifecycle / “AI” subsystem

  • NPCSpawner instantiates NPCs and injects route/table data.
  • Ren_Waypoint moves NPCs along a waypoint array (entry route) until the last waypoint and then enables dialogue.
  • EatAndLeave moves NPCs from “dialogue area → table route” and later from “table → exit route”, controls eating animation, and triggers tip + despawn.

B. Dialogue & order definition subsystem

  • Ren_Order selects a random OrderData script, plays a dialogue sequence (DialogueLine[]), triggers animations/audio, then:
    • tells the till what the correct order is (till.PlaceOrder(foodID, drinkID)), and
    • tells the NPC/table what should be served (GetComponent<EatAndLeave>().SetFoodAndDrink(foodID, drinkID)).

C. UI / ordering / economy subsystem

  • Till is both:
    • the UI controller (buttons, order screen open/close, item selection), and
    • the “economy” accumulator (earned money, tips), and
    • the validator of gameplay rules (checks if selected items match expected IDs).

D. World objects subsystem

  • Table is an in-world domain object: holds expected IDs, references where food/drinks are placed, and cleans up after eating.
  • Door + DoorTrigger implement a simple physical trigger that opens a door and plays an audio clip when an NPC enters.

Best-practice note: This decomposition is good because it gives you “chunks” to reason about. A common step up in architecture maturity is making each subsystem communicate through explicit interfaces or events, rather than locating each other dynamically.

2) The runtime flow (who calls whom, in what order)

Here’s the actual control-flow implied by the fetched scripts:

  1. Spawn and initialise NPC
    • NPCSpawner.SpawnRandomNPC() instantiates a random prefab.
    • It sets:
      • npc.Ren_Waypoint.waypoints = startWaypoints
      • npc.EatAndLeave.tableWaypoints/exitWaypoints/table = random TableData
    • This is “dependency injection” but done manually at runtime.
  2. Move to dialogue
    • Ren_Waypoint.Update() calls MoveToWaypoint() until the last waypoint.
    • On last waypoint, it stops walking and activates ActivateDialogue once.
  3. Dialogue and order setup
    • Ren_Order.Start() selects random OrderData, loads its DialogueLine[], and wires the talk button to AdvanceDialogue.
    • When dialogue ends:
      • till.ActivateOrderButton()
      • till.PlaceOrder(foodID, drinkID) (sets expected IDs internally in Till)
      • EatAndLeave.SetFoodAndDrink(foodID, drinkID) (stores expected IDs on the Table, and starts tip-timer logic)
      • hides dialogue UI
  4. Player uses till
    • Till builds an item catalogue mapping button GameObject names → Item(name, price, orderID).
    • Player picks up to 2 items; CalculateAndShowTotal() checks correctness against orderedFood/orderedDrink.
    • If correct:
      • add earned money
      • FindObjectOfType<EatAndLeave>()?.MoveToTable() to start table movement
    • If wrong:
      • FindObjectOfType<Ren_Order>().OrderIncorrect() plays feedback
  5. NPC sits, waits for food/drink objects, eats, tips, exits
    • EatAndLeave.MoveToWaypoint(tableWaypoints) until last; triggers sit.
    • It checks table.foodOnTable && table.drinkOnTable before starting the eating coroutine.
    • After eatTime, it:
      • marks exiting and moves along exit waypoints
      • calls table.CleanTable()
      • calls FindAnyObjectByType<Till>().Tip(tipAmount)
    • On reaching final exit waypoint:
      • NPCSpawner.instance.SpawnRandomNPC()
      • Destroy(gameObject)
  6. Door
    • DoorTrigger.OnTriggerEnter(Collider other) checks tag "NPC", calls door.OpenDoor(), and plays the door’s audio.
    • Door.OpenDoor() starts a coroutine to open for ~2s, then close, with trigger cooldown.

Best-practice note: This flow works, but a lot of it is held together by scene searches (FindObjectOfType) and global state (NPCSpawner.instance), which becomes fragile as the project grows or when multiple NPCs exist simultaneously.

3) Architectural strengths and weaknesses (teaching points)

Strengths (good student-friendly patterns present)

  • Component-based separation: movement (Ren_Waypoint, EatAndLeave) is separate from UI (Till) and from dialogue (Ren_Order).
  • Serializable data containers: OrderData and DialogueLine are serializable and can be configured in the Inspector—good for content-driven design.
  • Waypoint movement abstraction: both Ren_Waypoint and EatAndLeave implement waypoint traversal cleanly enough for small projects.

Weaknesses / risks (and why they matter)

  1. Tight coupling via scene searches
    • Till.CalculateAndShowTotal() uses FindObjectOfType<EatAndLeave>() and FindObjectOfType<Ren_Order>().
    • EatAndLeave uses FindAnyObjectByType<Till>().
    • This works for prototypes but becomes unreliable when:
      • there are multiple NPCs,
      • multiple tills/UI objects,
      • objects are disabled/enabled across scenes,
      • or ordering is more complex.
  2. Global singleton NPCSpawner.instance
    • Used by EatAndLeave to spawn the next NPC.
    • This is convenient, but introduces hidden dependencies and makes testing harder.
  3. Till mixes UI, domain logic, and orchestration
    • Till controls buttons/screens, validates correctness, updates economy totals, and tells NPCs to move.
    • Best practice is to keep UI display logic separate from gameplay rules and from NPC control.
  4. Implicit single-customer assumption
    • The code structure strongly suggests “one active NPC at a time”.
    • If later you want a queue or multiple tables active concurrently, you’ll need to refactor how you identify “which NPC is ordering” rather than using “find any NPC”.

Suggestions / refactoring improvements (practical and best-practice aligned)

A) Reduce coupling: replace scene searches with explicit references/events

Problem: FindObjectOfType picks an arbitrary instance and hides dependencies.

Refactor direction:

  • When the NPC finishes dialogue, it can register itself as the “current customer” in a controller (or in the till).
  • Or use C# events:
    • Ren_Order raises OnOrderReady(foodID, drinkID, customerRef)
    • Till subscribes and stores currentCustomer
    • When the order is correct, Till calls currentCustomer.MoveToTable()

This makes the system deterministic and scalable.

B) Split Till into smaller responsibilities (MVC/MVP style)

Right now Till is doing too much. A student-friendly refactor:

  • TillView (MonoBehaviour): owns Buttons/Text, emits UI events.
  • TillController (MonoBehaviour or plain C#): holds selected items, validates orders, updates money/tips.
  • OrderValidator / MenuCatalogue (plain C# or ScriptableObject): maps IDs to prices/names; computes totals.

This improves maintainability and testability.

C) Replace “button GameObject name → item” mapping with IDs or ScriptableObjects

Till.InitialiseItemsCatalogue() uses foodOptionButtons[i].gameObject.name as the key. That’s fragile.

Better options:

  • store an ItemButton component on each button with an itemId field, or
  • use ScriptableObjects: MenuItemSO { id, displayName, price }.

D) Make NPC state explicit (finite state machine)

EatAndLeave currently uses booleans like canMove, exiting, canEat, canStartTipTimer.

A clearer architecture is an enum:

  • CustomerState = Arriving, Ordering, WalkingToTable, WaitingForFood, Eating, Leaving

This reduces “boolean explosion” and clarifies transitions.

E) Improve robustness of Table.CleanTable()

CleanTable() destroys foodOnTable and drinkOnTable without null checks, and always calls bewerageMaker.ResetBeverageMaker().

Best practice:

  • null-check before destroying
  • only reset beverage maker if it exists
  • consider object pooling for spawned food/drink objects in larger projects.

F) Future scalability: move “spawn next NPC” responsibility out of EatAndLeave

Currently, EatAndLeave decides when to spawn the next NPC. Architecturally, spawning is a “game progression/system” concern.

Better:

  • EatAndLeave raises an event OnCustomerExited
  • NPCSpawner (or a CafeSessionController) listens and spawns next NPC.

This keeps responsibilities clean.

Final takeaway (architecture in one sentence)

Your project is a solid Unity prototype built around MonoBehaviour components cooperating through direct calls, scene lookups, and a spawner singleton; for best practice and scalability, the next step is to reduce coupling (events/explicit references), separate UI from domain logic, and make NPC/customer flow a more explicit state-driven pipeline.
