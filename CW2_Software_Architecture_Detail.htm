1) Overall Functionality (Architecture-Level Summary)

At a high level, this Unity project implements a single-customer gameplay loop for a cafe:

  1. Spawn a customer (NPC) at a spawn point (NPCSpawner).
  2. The NPC walks along entry waypoints to a dialogue location (Ren_Waypoint).
  3. The NPC plays a dialogue sequence (text, optional audio, animation), then sets expected foodID and drinkID (Ren_Order).
  4. The Till UI lets the player select exactly two items (one food, one drink), then validates the selected pair against expected IDs (Till).
  5. If correct, the NPC is instructed to walk to table and sit, then later eat and leave, possibly tipping the player (EatAndLeave + Till.Tip(...)).
  6. When leaving, the NPC triggers spawning the next NPC, then destroys itself (EatAndLeave calls NPCSpawner.instance.SpawnRandomNPC()).
  7. Tables can accept food/drink placements and be cleaned/reset (Table), and doors can open via trigger collisions (Door, DoorTrigger).

From a software architecture viewpoint, this is a scene-driven, component-based architecture typical of Unity projects, where MonoBehaviours act as both controllers and domain objects, and gameplay is coordinated through:

  • direct component references (GetComponent, public fields),
  • scene queries (FindObjectOfType, FindAnyObjectByType),
  • a global-like singleton (NPCSpawner.instance).

2) Main Subsystems (Separation of Concerns)

You can understand this project as a small set of cooperating subsystems:

A) NPC lifecycle / AI subsystem

  • NPCSpawner instantiates NPCs and injects route/table data.
  • Ren_Waypoint moves NPCs along entry waypoints until the last point, then enables dialogue.
  • EatAndLeave handles movement from dialogue to table to exit, eating animation, tip trigger, and despawn.

B) Dialogue and order-definition subsystem

  • Ren_Order selects random OrderData, plays DialogueLine[] sequence, triggers animations and audio.
  • Then it tells Till the correct order via PlaceOrder(foodID, drinkID).
  • It also sets expected serve values on the NPC/table via EatAndLeave.SetFoodAndDrink(foodID, drinkID).

C) UI / ordering / economy subsystem

  • Till is both UI controller (buttons, order screen open/close, item selection) and gameplay validator.
  • Till also accumulates economy values (earned money and tips).

D) World objects subsystem

  • Table stores expected IDs, placement references, and handles cleanup after eating.
  • Door and DoorTrigger implement trigger-based door opening with audio feedback.

Best-practice note: decomposition into subsystems is a strong foundation. The next architectural step is communicating through explicit interfaces/events instead of dynamic scene lookups.

3) Runtime Control Flow (Call Order)
  1. Spawn and initialize NPC
    • NPCSpawner.SpawnRandomNPC() instantiates a random prefab.
    • It injects Ren_Waypoint.waypoints, and EatAndLeave table/exit data from selected TableData.
    • This is manual runtime dependency injection.
  2. Move to dialogue
    • Ren_Waypoint.Update() repeatedly calls MoveToWaypoint() until the final waypoint.
    • At final waypoint, walking stops and ActivateDialogue is triggered once.
  3. Dialogue and order setup
    • Ren_Order.Start() picks random OrderData and wires talk button to AdvanceDialogue.
    • At end of dialogue: ActivateOrderButton(), PlaceOrder(foodID, drinkID), SetFoodAndDrink(foodID, drinkID), hide dialogue UI.
  4. Player uses till
    • Till maps button object names to Item(name, price, orderID).
    • Player picks up to 2 items. CalculateAndShowTotal() checks against orderedFood/orderedDrink.
    • If correct: add earned money and trigger MoveToTable() on EatAndLeave.
    • If wrong: call Ren_Order.OrderIncorrect() for feedback.
  5. NPC sits, eats, tips, exits
    • EatAndLeave moves along table waypoints and triggers sit behavior.
    • After both table.foodOnTable and table.drinkOnTable are present, eating starts.
    • After eatTime: mark exit, move to exit waypoints, clean table, apply tip via Till.Tip(tipAmount).
    • At final exit waypoint: spawn next NPC and destroy current NPC GameObject.
  6. Door interactions
    • DoorTrigger.OnTriggerEnter checks for tag NPC, then opens door and plays audio.
    • Door.OpenDoor() opens for roughly 2 seconds, then closes with cooldown.

Best-practice note: this flow is functional, but it depends heavily on scene searches and global state, which becomes fragile as systems grow or multiple NPCs run concurrently.

4) Strengths and Weaknesses

Strengths (student-friendly patterns already present)

  • Component-based separation: movement (Ren_Waypoint, EatAndLeave), UI (Till), and dialogue (Ren_Order) are split.
  • Serializable data containers: OrderData and DialogueLine are inspector-configurable and support content-driven design.
  • Waypoint movement abstraction: waypoint traversal is reused and understandable for small projects.

Weaknesses and risks (and why they matter)

  1. Tight coupling via scene searches
    • Till uses FindObjectOfType() and FindObjectOfType().
    • EatAndLeave uses FindAnyObjectByType().
    • Fragile when multiple NPCs/tills exist, objects are toggled, or ordering flow grows.
  2. Global singleton NPCSpawner.instance
    • Convenient but introduces hidden dependencies and harder testing.
  3. Till mixes concerns
    • UI control, validation logic, economy accumulation, and NPC orchestration are in one class.
  4. Implicit single-customer assumption
    • Current structure strongly assumes one active customer at a time.
    • Scaling to multiple simultaneous customers/tables requires order ownership refactoring.

5) Refactoring Improvements and Final Takeaway

Practical, best-practice-aligned improvements

A) Reduce coupling: replace scene searches with explicit references or events

  • Problem: FindObjectOfType picks arbitrary instances and hides dependencies.
  • Refactor direction:
    • When dialogue ends, register that NPC as current customer in a controller or till.
    • Or use events: Ren_Order raises OnOrderReady(foodID, drinkID, customerRef), Till subscribes and stores currentCustomer, then calls currentCustomer.MoveToTable() if validated.
  • Outcome: deterministic and scalable flow.

B) Split Till into smaller responsibilities (MVC/MVP style)

  • TillView: owns buttons/text, emits UI events.
  • TillController: stores selected items, validates orders, updates money/tips.
  • OrderValidator/MenuCatalogue: maps IDs to price/name and computes totals.

C) Replace button-name mapping with IDs or ScriptableObjects

  • Current gameObject.name mapping is fragile.
  • Use ItemButton component with itemId, or MenuItemSO assets with id/displayName/price.

D) Make NPC state explicit with a finite state machine

  • Current booleans can cause complexity growth.
  • Use enum state model: Arriving, Ordering, WalkingToTable, WaitingForFood, Eating, Leaving.

E) Improve Table.CleanTable() robustness

  • Add null checks before destroy calls.
  • Reset beverage maker only if instance exists.
  • Consider object pooling for larger spawned-object counts.

F) Move spawn-next responsibility out of EatAndLeave

  • EatAndLeave should raise OnCustomerExited.
  • NPCSpawner or CafeSessionController should handle next spawn.

Final takeaway

This project is a strong Unity prototype built on MonoBehaviour components with direct calls, scene lookups, and a spawner singleton. For better scalability, reduce coupling through events or explicit references, separate UI from domain logic, and drive NPC/customer behavior through an explicit state pipeline.
