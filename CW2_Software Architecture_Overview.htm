1) Overall Functionality (Software Architecture Viewpoint)

This project is a Unity cafe simulation loop built around:

  • A central game-state/controller: CafeManager
    • Tracks completion flags like restocking, shop open, back door state, music, and more.
    • Triggers progression (for example, after tasks are done, spawn NPCs or open a door).
  • Interaction as input events on scene objects: InteractionHandler
    • Unity style: clicking objects (OnMouseDown) triggers actions by calling CafeManager.ToggleTaskCompletion(...) or similar.
  • NPC lifecycle pipeline
    • NPCSpawner (singleton-like via static instance) spawns random NPC prefabs and injects waypoint routes and table assignments via TableData.
    • Ren_Waypoint moves NPCs along waypoint routes to interaction points/tables and can enable dialogue.
    • Ren_Order runs an order dialogue sequence (text, audio, animation states) and connects to the till for ordering.
    • EatAndLeave handles the consume at table to leave to possibly spawn next NPC loop, and awards tips.
  • Ordering and UI orchestration: Till
    • Owns UI widgets (buttons, text fields) and order-screen state flow (open/close, add items, confirm/cancel).
    • Calculates totals and updates earned money and earned tips.
  • Table as an in-world domain object: Table
    • Stores expected/placed food-drink IDs, placement transforms, and supports clean/reset.
    • Resets external systems like BewerageMaker.
  • Door subsystem
    • Door animates or moves a door.
    • DoorTrigger listens for OnTriggerEnter and opens the door when an NPC enters.

Architectural style (as implied by the diagram)

  • Predominantly MonoBehaviour-driven, object-oriented, with runtime references between components.
  • A blend of GameObject composition (Unity components) and manager orchestration (CafeManager, NPCSpawner, Till).
  • Gameplay is mostly event-driven (clicks, triggers, button presses), but implemented through direct method calls between components.

2) Subsystems and Best-Practice Framing

A) Identify subsystems (good architectural habit)

A clean way to reason about this project is to split it into subsystems:

  1. Game State / Progression
    • CafeManager
    • Responsibility: maintain what tasks are done and decide what can happen next.
  2. Interaction/Input
    • InteractionHandler
    • Responsibility: translate player actions (click) into domain actions (for example, restock completed or back door opened).
  3. NPC / AI Flow
    • NPCSpawner, Ren_Waypoint, EatAndLeave
    • Responsibility: create NPCs, route them, and manage lifecycle.
  4. Narrative / Dialogue and Feedback
    • Ren_Order, DialogueLine, OrderData
    • Responsibility: show lines, play audio, set animation states, and advance dialogue.
  5. Ordering / Economy / UI
    • Till, Item
    • Responsibility: UI flow, pricing, and payouts.
  6. World Objects
    • Table, Door, DoorTrigger (plus referenced BackDoorAnim, BewerageMaker)
    • Responsibility: in-world state and animations.

This is a software-architecture lens: what changes together belongs together. If you change UI, you ideally should not break NPC movement.

3) Key Architectural Relationships and Risks

B) Key architectural relationships (and implications)

1) CafeManager as a God-object risk

The diagram shows many systems calling into or being triggered by CafeManager (interaction, restock, spawner, back-door animation). This is common in student Unity projects.

Best practice idea: keep CafeManager as a state holder plus coordinator, but avoid making it the place where everything happens. A single oversized class becomes hard to test, hard to reuse, and hard to understand.

2) Direct references create tight coupling

Examples from the diagram:

  • InteractionHandler -> CafeManager
  • Ren_Order -> Till
  • EatAndLeave -> NPCSpawner (spawn another NPC)
  • Till -> EatAndLeave (tell NPC to move to table)
  • Table -> BewerageMaker

This is tight coupling: changing one class often forces changes in another.

Best practice: introduce interfaces or an event system for communication.

  • UI can raise OrderConfirmed(foodId, drinkId).
  • The active NPC can subscribe if it is the current customer.
  • Or a mediator/controller can coordinate.

3) UI and domain logic appear mixed in Till

Till owns buttons/text fields, but also pricing, item lists, tips, totals, and NPC triggers.

Best practice separation:

  • View (UI): shows/hides screens and captures button clicks.
  • Controller/Presenter: processes UI events and updates view.
  • Domain (Model): order calculation, pricing rules, validation.

4) Data structures (OrderData, DialogueLine, Item) are good signs

These are simple data carriers and a step toward cleaner architecture. They can evolve into ScriptableObjects, serializable DTOs, or testable logic inputs.

4) Cleaner Target Architecture (Teachable Version)

If explaining an idealized architecture for this project:

  • Core domain layer (pure C# where possible)
    • Order, MenuItem, PriceCalculator, DialogueSequence.
    • No MonoBehaviour and no direct Unity UI references.
  • Application layer (game rules)
    • CafeSessionController coordinates tasks, NPC queue, and order flow.
    • Emits events such as OnCustomerArrived, OnOrderTaken, OnOrderServed.
  • Presentation layer
    • TillView (buttons/text), DialogueView.
    • Subscribes to controller events and forwards button clicks to controller.
  • Infrastructure layer (Unity specifics)
    • NPCSpawner, movement components, audio, and animation.
    • Adapters that translate controller intent into Unity actions.

You do not have to implement full Clean Architecture, but thinking in layers helps keep code maintainable.

5) Refactoring Opportunities

High-impact refactors

  1. Replace NPCSpawner.instance with dependency injection or scene references
    • Static instance is quick, but encourages global access.
    • Better: assign references in the Inspector, or use a single GameContext object for shared services.
  2. Introduce an event bus (UnityEvent or C# events)
    • Instead of Till -> EatAndLeave direct call:
    • Till raises OrderConfirmed(foodId, drinkId), then the active NPC or a controller responds.
    • This reduces bidirectional coupling.
  3. Split Till into UI and logic
    • TillView (buttons/text)
    • TillController (order-screen flow, calls services)
    • OrderService or OrderCalculator (price, totals, validation)
    • This is easier to test and reason about.
  4. Use ScriptableObjects for menu items and dialogue
    • Item could become MenuItemSO (name, price, id, prefab references).
    • OrderData could be a ScriptableObject asset for customer script/dialogue.

Medium-impact improvements

  1. Define explicit NPC state machines
    • NPC scripts often become hard to manage with many booleans.
    • Use an enum like CustomerState = Arriving, Ordering, WaitingForFood, Eating, Leaving.
    • Or use a small state-pattern approach.
  2. Avoid stringly-typed task toggles
    • ToggleTaskCompletion(taskName: string) is fragile and typo-prone.
    • Use a TaskType enum or explicit methods/events.
  3. Clarify ownership of money/tips
    • Till appears to track earnings and tips.
    • Consider a Wallet/EconomyManager that owns totals, while UI displays them.
